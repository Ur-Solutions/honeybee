# Accounts Model — Implementation Spec for S1–S3

Scope: worktree `/Users/trmd/Projects/trmd/honeybee/worktrees/honeybee/accounts-model`. The design doc (`docs/accounts-model.md`) is authoritative: account = `(cli, model-provider)` pair, full per-account credential isolation, profiles are thin sugar. Stored field name `tool` is kept and aliased as `cli` for back-compat; `provider` is new+required; `model` is optional. Single-provider CLIs get a canonical provider (claude→anthropic, codex→openai, grok→xai, kimi→moonshot).

Throughout, "keep suite green / no behavior change" is a hard constraint for S1 and the limits refactor in S3.

2. `isAccountRecord` (lines 73–82) — **must stay permissive** so `loadAccounts` never drops legacy entries. Keep the existing required-field check (id/tool/label/addedAt). Do NOT add `provider` to the type guard. Instead introduce a normalizer that backfills on read:
```ts
const PROVIDER_BY_CLI: Record<string,string> = {
  claude:"anthropic", codex:"openai", grok:"xai", kimi:"moonshot",

  if (a.provider) return a;

export type AddAccountOptions = { email?: string; provider?: string; model?: string };

Resolve `provider = options.provider ?? PROVIDER_BY_CLI[kind]`; throw a clear error if still undefined (forces `--provider` for opencode). Write `provider` (+`model` if set) into the record. Ledger `account.add` event — add `provider` field.

5. `removeAccount` ledger (line 124) — add `provider: account.provider` for audit symmetry (optional, cheap).

## S2 — Account-first spawn (keystone)

Goal: `hive spawn <account>` (and profile→account) resolves cli + provider + model + an **isolated home**, and **always** sets `record.accountId`. This supersedes the config `kind` alias as the account-binding mechanism (the alias stays as internal CLI plumbing until S5).

### Resolution chain

**`src/accounts.ts` — `resolveSpawnAgent(kind)` (lines 187–198):** today it returns `{agent}` for any token that is a driver kind, only binding an account via `<tool>-<query>` shorthand. Add an **account-first** branch *before* the driver-kind passthrough:

```ts
export async function resolveSpawnAgent(kind: string): Promise<SpawnAgentSpec> {
  // 1. exact account id/label (account-primary): `hive spawn minimax`, `hive spawn claude-ursolutions`
  const direct = await tryFindAccountExact(kind);     // new: exact-only, no fuzzy, no throw
  if (direct) return { agent: direct.tool, account: direct };
  // 2. plain driver kind passthrough (claude, cc1, codex2) — unchanged
  if (hasAgentDriver(canonicalAgentKind(kind).toLowerCase())) return { agent: kind };
  // 3. <tool>-<query> shorthand — unchanged
  ...
}
```
`tryFindAccountExact` must be **exact id/label only** (reuse the exact arm of `matchAccount`) to avoid stealing plain `claude`/`codex` tokens via fuzzy match. Ordering matters: a token that is simultaneously a valid driver kind and an account id resolves as the account (desired — that is the keystone). A token like `claude` is not an account id (ids are `claude-<label>`) so it still passes through as a plain kind.

### Carrying provider+model into the spawn command

The CLI adapter needs the account's `model` (and, for opencode, `provider`) to build `--model <provider>/<model>`. Two touch points:

**`src/cli.ts` — `spawnBee` (lines 342–393):**
- `home` selection (line 346) already does `opts.home ?? defaultHomeForAccount(opts.account)` — this is the isolated home. Keep. The isolated store is `~/.hive/homes/<account.id>/`. No change needed for isolation itself; what changes is that account-first spawns now reach this path for *every* provider (minimax/glm/kimi/grok), not just claude/codex.
- `resolveAgent(opts.agent, …, { identity: Boolean(opts.account) })` (line 347) — extend to pass the account's model so the command can embed it. Add `model` to `ResolveAgentOptions` and to `SpawnOptions`; `spawnSingleBee` sets `opts.model = account?.model`.
- `record.accountId` (line 387) — already set when `opts.account`. The keystone requires that account-first spawns **always** have `opts.account`, which `resolveSpawnAgent` branch 1 now guarantees. No new `accountId` plumbing — the existing gate becomes universally satisfied for account spawns.

**`src/agents.ts` — `resolveAgent` (lines 82–122):**
- Add `model?: string` to `ResolveAgentOptions` (line 69). When set and no explicit command override exists, append the CLI-adapter's model selector. The selector form is CLI-specific (doc: opencode `--model <provider>/<model>`; single-provider CLIs implicit). Implement as a driver hook rather than inline branching:

**`src/drivers.ts` — add `modelArgs?(model, provider): string[]` to `AgentDriver`:**
```ts
opencode: { …, modelArgs: (model, provider) => model ? ["--model", `${provider}/${model}`] : [] },
claude:   { …, modelArgs: (model) => model ? ["--model", model] : [] },
codex:    { …, modelArgs: (model) => model ? ["--model", model] : [] },
// grok/kimi: implicit or per-CLI; start with [] and refine in S4
```
`resolveAgent` calls `agentDriver(profile.kind)?.modelArgs?.(options.model, providerForSpawn)` and concatenates into `args` (before `extraArgs` so user `-- …` still overrides). Provider passed in must come from the account; thread it via a new `ResolveAgentOptions.provider`. For non-account spawns both are undefined → `modelArgs` returns `[]` → **no behavior change**.

### Thin profiles → accounts

type BeeConfig = { yolo?; home?; command?; kind?;   // existing (kind kept for back-compat)

Precedence: explicit CLI flag > profile > account default. This is where a profile "references an account."

- Today `bees.minimax.kind="opencode"` makes `canonicalAgentKind("minimax")→"opencode"` (agents.ts `profileAlias` lines 138–151) but yields **no account** → no `accountId` → no usage (the exact bug in the doc, lines 18–20). 
- After S2, `bees.minimax = { account: "minimax", model: "MiniMax-M3" }` resolves to the `minimax` account → `accountId` always set. The `kind` alias is no longer needed to pick the driver: the account's `cli` (`tool`) field supplies the driver. `profileAlias`'s config branch (lines 148–149) stays as a fallback for un-migrated configs but is dead for account-backed profiles. Removal deferred to S5.

## Risks & open questions

- **opencode XDG isolation env.** Driver `extraEnv: { XDG_DATA_HOME: "{home}/xdg-data" }` (drivers.ts 89) is the *only* thing isolating opencode auth. The maps do not confirm that two opencode accounts (minimax, glm) with distinct `~/.hive/homes/<id>/xdg-data` actually get separate `auth.json` blobs at runtime — `OPENCODE_CONFIG_DIR` is set too (homeEnv) but config vs data split is untested. **Must verify in S2/S4 that XDG_DATA_HOME alone fully isolates the login**, otherwise full per-account isolation (settled decision #1) is not achieved. This is the highest-risk unknown.
- **`model` → command selection for grok/kimi.** Doc only specifies opencode (`--model <provider>/<model>`) and "implicit" for single-provider CLIs. grok/kimi `modelArgs` start as `[]`; whether they accept a `--model` flag is unverified — left for S4 with real CLIs.
- **`opencode-opencode1` retire.** Per the user this is retired in S4, not S1. Until then it loads provider-less and is excluded from provider-keyed features. If any existing test asserts limits behavior for it, that test must be updated/removed — flag during S1.5.
- **S4 provider re-logins are pending (user).** minimax/glm accounts cannot be fully exercised end-to-end (real `fetchLimits` against live z.ai/minimax endpoints) until the user re-logs each opencode provider into its isolated store. S3 fetchers should be tested against mocked HTTP only; live verification is an S4 gate. Endpoint auth schemes (z.ai Bearer OAuth vs minimax API-key) are from the doc and unverified against live responses — response-shape parsing may need adjustment after first real call.
- **Transcript provider string for opencode.** `transcriptTokenTotals(transcript.provider, …)` (usageSampler 96) — the maps don't enumerate what `provider` string opencode transcripts carry, nor whether opencode emits a honeybee-readable transcript at all (`hasTranscriptProvider: true` for opencode, but token-extraction branch unconfirmed). If absent, usage sampling for opencode bees yields exhaustion events but no token samples — acceptable but worth noting.
- **kimi has `hasTranscriptProvider: false`** → kimi bees get exhaustion edges but never token samples; consistent with current behavior, not a regression.
- **AccountId field rename deferred.** `tool` is kept as the on-disk cli field through S5. Any external tooling reading `accounts.json` and switching on `tool` keeps working; `provider` is purely additive. If S5 ever moves vault paths to include provider, a real disk migration (with the lazy-on-touch strategy) is required — out of scope here.

# Adversarial Completeness Review — Accounts Model S1–S3

## CRITICAL (breaks build/tests/back-compat)

**1. `resolveAgent` is synchronous, but S2.3 routes account resolution through it in spawnBee — fine; however `resolveAgent` signature change adds `provider`/`model` but `spawnBeeForFlow` (agents.ts:269-271) does NOT thread them and has NO `account` field in `SpawnBeeOptions`.** The spec's table claims "thread model/provider/account so flow-spawned bees are account-bound too," but `spawnBeeForFlow` never calls `activateAccountIntoHome` and never sets `record.accountId` (agents.ts:287-309 has no `accountId`). Account-binding for flow spawns is a *new capability*, not a thread-through — it requires adding `account` to `SpawnBeeOptions`, an activation call, and `accountId` in the record. The spec under-scopes this to "thread model/provider." Fix: S2.2/S2.4 must add `account?: AccountRecord` to `SpawnBeeOptions`, call `activateAccountIntoHome`, and set `record.accountId` in `spawnBeeForFlow` — or explicitly defer flow account-binding and drop it from the S2 test plan.

**2. `tryFindAccountExact` ordering hazard: branch 1 runs BEFORE branch 2, but `matchAccount`'s "exact" arm matches on `account.label`, not just `account.id`.** A user with an account labeled `cc1`, `codex`, `grok2`, or `claude` (labels are free-form, e.g. someone logs in `hive account add claude claude`) would have branch 1 steal that token away from the driver-kind passthrough. The spec asserts "a token like `claude` is not an account id (ids are `claude-<label>`)" — true for *ids*, but `matchAccount` also matches `label`, and label can be exactly `claude`/`cc1`/`codex2`. Fix: `tryFindAccountExact` must match on `account.id` ONLY (not label), or explicitly exclude tokens where `hasAgentDriver(canonicalAgentKind(kind))` is true before the exact lookup.

**3. `account.tool === "claude"` literal at limits.ts:109 (catch-block source default) AND the daemon chain-sync filter are not the only claude-keyed sites the registry refactor must touch.** The spec re-keys limits.ts:109 and run.ts chain-sync, but misses several `account.tool === "claude"` sites in **accounts.ts itself** that are load-bearing for activation and remain tool-keyed (correctly, per the doc — claude OAuth is anthropic-only): `activateAccountIntoHome` (lines ~285, ~331), `captureAccountFromHome` (line ~240), and `runLoginSeat`/`cmdAccountCapture` in cli.ts (~4171, ~4195). These are fine to leave tool-keyed, BUT the spec's claim that S1 is "no behavior change" while `normalizeAccountRecord` backfills `provider` means these sites now coexist with a `provider` field that they ignore. No bug, but the spec never states the invariant "claude-OAuth machinery stays keyed on `tool==='claude'`, NOT `provider==='anthropic'`" — except for run.ts which it *does* re-key. Inconsistency: run.ts re-keyed to `provider==='anthropic'` but accounts.ts activation stays `tool==='claude'`. Fix: state explicitly that activation/capture/keychain stay `tool`-keyed; only limits dispatch + chain-sync filter move to provider. Re-keying run.ts but not accounts.ts is arbitrary and the test plan doesn't cover it.

## HIGH (gaps / missed call-sites)

**4. `resumeArgs(tool, providerSessionId)` in swap.ts:116-121 is tool-keyed and the spec never addresses model re-injection on swap/resume.** When `swapAccount` resumes via `resolveAgent(record.requestedAgent ?? record.agent, resumeArgs(...), {home, yolo, identity:true})` (swap.ts:81), it does NOT pass `model`/`provider`. After S2 makes spawns carry `--model <provider>/<model>`, a swapped opencode bee resumes WITHOUT its model selector — the resumed command drops the model. The spec's S2 `modelArgs` design and S3 autoswap both touch swap.ts but neither threads model into the resume `resolveAgent` call. Fix: swap.ts:81 must pass `{ model: account.model, provider: account.provider }` (the new account's model after swap) so resumed bees keep model selection.

**5. opencode `modelArgs` collides with `DEFAULT_COMMANDS.opencode = "opencode run --interactive"` and config-driven `command` overrides.** For migrated profiles, the doc example was `command: "opencode run --interactive --model minimax/MiniMax-M2"` (config.ts BeeConfig). If S2 makes the account supply `model` AND a legacy config still has `--model …` baked into `command`, `resolveAgent` will append a SECOND `--model` (commandOverride at agents.ts:95 wins for the base command, then `modelArgs` concatenates). Double `--model` flag. Same risk for kimi/grok whose `DEFAULT_COMMANDS` already embed flags. Fix: `modelArgs` must be suppressed when `commandOverride` is present (i.e., when the resolved command came from config/env, not the driver default), OR the profile→account migration must strip `--model` from `command`. The spec's precedence note ("before extraArgs so user `-- …` overrides") doesn't cover the command-embedded-model case.

**6. `resolveAccountFlag(query, tool, ...)` and `findAccount(query, tool)` scope by `account.tool === canonicalAgentKind(tool)` — provider is never a filter.** Once opencode hosts minimax+glm+kimi, `hive spawn opencode --account minimax` and `<tool>-<query>` shorthand resolve within ALL opencode accounts regardless of provider; `pickAutoAccount(tool)` → `pickLeastLoadedAccount` (limits.ts:770-792) filters by `canonicalAgentKind` only. The spec narrows *autoswap* candidates by provider but leaves `findAccount`/`pickAutoAccount`/`resolveAccountFlag` provider-blind, so auto-pick could select a glm account for a token the user meant as minimax. Fix: either accept that account-first resolution (exact id) sidesteps this, or add provider scoping to `pickLeastLoadedAccount` and `findAccount` when multiple providers share a cli. At minimum flag that `<tool>-auto` is provider-ambiguous for opencode.

**7. `transcriptTokenTotals` returning `null` for opencode is safe at usageSampler.ts:96, but the spec's "add an opencode branch" assumes opencode transcripts carry a usable provider string.** usageSampler reads `transcript.provider` from `readRows(record)`. The map flags this is unconfirmed. If `readRows` returns `provider: "opencode"` but the rows have no extractable usage, `null` is fine (no crash). But if `readRows` *throws* for opencode (no transcript provider wired), the `.catch(() => null)` at line 95 already handles it. So this is actually SAFE as-is — the spec over-worries here but should confirm `hasTranscriptProvider("opencode")` is true (it is, per drivers map) so `readRows` is attempted. No code change strictly required for null-safety; the spec's "or return null gracefully" is already the behavior. Minor: don't block S3 on this.

## MEDIUM (correctness / clarity)

**8. `AccountLimits.tool` field is consumed by the table/cache and the spec keeps it, but `cachedAccountLimits` (limits.ts:669-691) keys by `account.id` — a legacy account whose `provider` is undefined (opencode-opencode1) will hit `providerAdapter(undefined)` → `undefined` → `unsupported`.** That matches today's behavior (good), BUT `accountLimits` now does `providerAdapter(account.provider)` where `account.provider` is `undefined` for the un-normalized opencode entry. `providerAdapter(undefined)` → `PROVIDERS[undefined]` → `undefined` — safe. However `listAccounts` backfill (S1.3) runs `normalizeAccountRecord` which leaves opencode provider-less, so by the time limits sees it, `account.provider` is `undefined` and the error string `${account.provider} has no limits source yet` prints "undefined has no limits source yet". Fix: guard the error message for missing provider; cosmetic but user-facing.

**9. `isAccountRecord` stays permissive (no `provider` check) — correct — but `normalizeAccountRecord` is applied in `listAccounts` ONLY.** Other readers that parse the registry directly bypass backfill. The spec asserts all reads go through `listAccounts`, but `findAccount` → `listAccounts` (OK), `removeAccount` → `listAccounts` (OK), `addAccount` → `listAccounts` (OK). Verify no raw `readFile(accountsRegistryPath())` exists elsewhere. Grep confirms `listAccounts` is the sole reader — SAFE. The spec is solid here; note only that `swapAccount` validation (`tool !== account.tool`) at swap.ts:32 receives accounts via callers, so the provider-match assertion the spec adds to swap.ts:31 must guard against `account.provider === undefined` (legacy) to avoid rejecting valid legacy swaps. Fix: provider-match assertion in swap.ts must skip when either side's provider is undefined (back-compat).

**10. Ordering hazard S2.1 → S2.4: adding `modelArgs` to the `AgentDriver` type (S2.1) and `ResolveAgentOptions.model/provider` (S2.2) before `resolveAgent` consumes them is fine, but S2.3 (`resolveSpawnAgent` account-first branch) makes `resolveSpawnAgent` ASYNC-returning an account for bare tokens — and `resolveSpawnAgentWithAuto` (cli.ts:413) is already async and wraps it. No build break. BUT the test fixture suite (`tests/*`) that asserts `resolveSpawnAgent("claude")` returns `{agent:"claude"}` with NO account will still pass only if branch 2 precedes any label match — see gap #2. Several existing tests likely call `resolveSpawnAgent` with a registry fixture present; if any fixture has a label colliding with a driver kind, S2.3 flips the result. Fix: audit existing `resolveSpawnAgent` tests' fixtures before landing S2.3.

**11. `spawnHomogeneousSwarm` (cli.ts:470, 478) gets `account` from `resolveSpawnAgentWithAuto` but the spec table says "same account/model threading" — yet swarm spawns N bees from ONE account.** All N bees would share one `accountId` and one isolated home (`defaultHomeForAccount(account)`), meaning N concurrent bees writing the same `xdg-data/opencode/auth.json` / same `~/.hive/homes/<id>`. This is a real isolation/concurrency hazard the spec doesn't flag: account-first swarm spawns put N processes in one home. Fix: either forbid `--count` with account-first bare tokens, or give each swarm bee a distinct home — the spec must decide, not silently thread.

## LOW (noted, not blocking)

**12. opencode XDG isolation (the spec's own "highest-risk unknown"):** confirmed the ONLY isolation is `extraEnv: { XDG_DATA_HOME: "{home}/xdg-data" }` (drivers.ts opencode) applied only when `identity:true`. `resolveAgent` merges it at agents.ts:111-113 *only if* `options.identity && profile.homePath`. Account-first spawns set `identity:true` via `Boolean(opts.account)` (spec confirms), so isolation env IS applied. The unverified part — whether opencode also reads `OPENCODE_CONFIG_DIR` for auth and whether two homes truly get separate `auth.json` — is genuinely untestable from code and correctly deferred. No correction; the spec's risk call is accurate.

**13. `account add` ledger event (`account.add`) adds `provider`, but `removeAccount` ledger (accounts.ts:124) is "optional, cheap" per spec — fine.** No issue.

**14. `AddAccountOptions` throw-for-opencode-without-provider (S1.2) changes `addAccount` behavior:** any existing test or script doing `addAccount("opencode", label)` with no provider now THROWS. The spec's test plan covers this as intended, but flag that `tests/*` and the `account login` flow (`cmdAccountLogin` may create an opencode account implicitly) must pass `--provider` or break. Fix: audit `runLoginSeat`/`cmdLogin` auto-create paths (cli.ts ~4048, ~4270) for opencode account creation without provider.

### Summary of must-fix before landing
- **#1** flow-spawn account-binding is new code, not a thread-through (or defer + drop from tests).
- **#2** `tryFindAccountExact` must match `id` only, not `label` (ordering keystone correctness).
- **#4** swap/resume drops model selector — thread `model`/`provider` into swap.ts:81.
- **#5** double `--model` when config `command` already embeds it.
- **#11** account-first `--count` swarm = N bees in one home (isolation break).
- **#3/#9** decide and document: claude machinery stays `tool`-keyed while limits/chain-sync go `provider`-keyed; provider-match in swap must tolerate undefined (legacy).
